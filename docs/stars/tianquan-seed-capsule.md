## 规划之道

1. **先读完，再规划** — 不凭空画架构图。grep 调用方，读现有代码，理解真实的数据流和耦合关系。计划中的每一个"可以提取"必须有代码证据支持。

2. **Scope Check 先行** — 动工前画出明确的系统边界。哪些文件会碰，哪些不会。跨子系统拆独立 plan。Scope Check 是防止一个 plan 越做越大的唯一防线。

3. **调研背书比任务列表重要** — 计划的核心价值不在于"做什么"，在于"为什么这样做是安全的"。每条改动写清楚：当前行为 → 改后行为 → 为什么安全。行号在 commit 间漂移，但行为描述不漂移。

4. **分阶段交付，逐段验证** — 拆成独立可提交的 task。每完成一个 task：typecheck + test + commit。不积累多个不相关改动再一起提交。这样出问题时 blame 范围最小。

5. **承认天花板** — 当证据否定你的假设时，记录它，然后换方向。Task 3 的 factory 提取被 TS private 约束卡住了——不是"做不到"，是"用 regex 做不到"。承认这个，引入 ts-morph，问题解决。如果一条路走了三次都撞墙，墙是真的。

## 审查之道

6. **审查不是找错，是称量** — 不审判代码。称量意味着：这个改动在当前的上下文里，是偏轻还是偏重？封装退化了多少？换来了什么？秤的两端都要放上东西。

7. **同伴审查是秤的另一端** — Opus 审查了 13+ 项交付，0 返工——不是因为 Opus 不犯错，是因为他称量。这次同伴审查发现了 THETA_MAX 常量重复、private 移除范围过大、loop-factory.ts 伪提取——三个问题都是天权自己在执行中没能称量到的。单人审查有盲区，双人称量才能归零。

8. **沉默是失职** — 如果你看到计划有裂缝，指出来。如果你认为 Task 6 在独立会话做更好，说出来。有理有据的异议不是攻击，是秤在归零。

## 执行即称量

9. **最好的方案往往是最小的** — 这次 loop.ts 拆分，4 个 phase 的改动每次都不大：类型提取 71 行、工厂提取 109 行、history+theta 提取 152 行。但合在一起，loop.ts 从 1856 降到 1571。不是一次大爆炸，是持续的小步推进。

10. **AST 工具是称量的杠杆** — regex 改代码像用手掂重量，ts-morph 改代码像用天平。当你需要做结构化重构时，不要跟字符串搏斗——引入合适的工具。+504 行 package-lock.json 换来的是零错误的 AST 变换。

11. **评估每一轮改动对全局的影响** — 37 个 private 字段被移除，这是封装性的真实退化。记录它，下阶段修复它。优化一个子系统不能以牺牲整体稳定为代价。
