# 对账式攻坚 — 复杂问题方法论胶囊（主题别名）

> 核心教训：行为事实只能从测量与对账获得，不能从想象获得。

<seed-capsule star="攻坚方法论" sealed="2026-07-17" gist="复杂问题攻坚的六条对账纪律：精确构成→测量先行→插桩对账→仿真回放→独立通道→失败产信息">
六条对账纪律。与诊断阶梯互补：阶梯回答"用什么手段逐级取证"，
本胶囊回答"证据如何构成才算数"。

① 确认精准构成再动手
   改任何代码前，先把它的精确构成算出来：公式、不变量、状态机迁移表。
   实例：输入框右角残缺，先把三行框线的宽度公式从装配代码里列出来
   （顶框 innerWidth+3 vs 输入行/底框 innerWidth+4）——off-by-one 在读
   第二遍代码时就定位了，不需要任何运行。算不清的地方就是理解缺口，
   先补理解，不先动代码。

② 测量先行，不凭空推理
   行为问题不要用想象回答。用既有测试夹具（harness）驱动真实组件，
   在关键参数轴上扫一遍（如列宽 80/60/40/26），打印实测值。
   实例：四种列宽实测证明三行框线宽度全相等——既证实了修复，也顺手
   证伪了三个想象中的候选根因。

③ 插桩对账
   包装目标函数（monkey-patch / 日志探针），记录它的实际行为值；
   期望值必须由独立路径推导（手工演算 / 规格 / 参考实现），两者逐帧
   对账，不一致即根因现场。实例：包装 buildFullRewrite 记录实际爬升量，
   与"上帧在新宽度下的 reflow 行数"逐帧对账——干净路径一次性排除，
   嫌疑直接落到 CPR 污染恢复路径。插桩脚本可留档沉淀为 scripts/ 探针，
   下次同场景直接复用。

④ 仿真回放
   环境与症状的交互太复杂时（终端 reflow、并发时序、缓存层级），造一个
   最小环境模型：只建模与症状相关的子集（擦除/折行/宽字形占位），
   确定性回放场景。仿真把"这个机制会不会产生这种症状"变成可执行的
   判定题，而不是观点题。

⑤ 期望值走独立通道
   对账的期望值绝不来自被测系统自身——否则是循环验证：系统错了，
   期望也跟着错，对账永远通过。期望值只能来自规格、手工推导、参考
   实现、或不变的物理约束（如"任何行宽 ≤ cols"）。

⑥ 失败必须产出信息
   每次探测结束，要么淘汰一条假设，要么收窄搜索范围——两者皆无就是
   无效探测，是预算浪费不是进展。连续两次无效探测 = 换手段，不换文案。
   （与 PAL 计分同源：uninformative 零分。）

PAL 桥接：attack_case 账本承载假设竞争，本胶囊指导探针构造——
plan_probe 优先选"对账型谓词"（command_output_matches 挂 ✓/✗ 输出、
test_outcome 挂最小仿真），让每个探针自带机器可判定的裁决面。
现成范例：scripts/tui-probe-*.ts（夹具测量 / 欢迎页形态 / resize 爬升对账）。

横切纪律：
- 排除法也是证据：干净路径被实验证明一致后，嫌疑收窄就是进展——
  不在已排除的方向上继续花费。
- 悖论即物理实验：两个"已验证事实"互相矛盾时，停止读更多代码，
  立即用 ②③④ 之一做物理裁决。
- 叙事引力不是证据：上下文里反复出现的框架天然显得更亮——频繁
  复现的嫌疑方向要过和其他方向一样的对账，不因眼熟而免审。
