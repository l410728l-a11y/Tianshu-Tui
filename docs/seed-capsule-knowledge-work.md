# 知识工作方法论 — 从杂乱素材到专家级成果

> 场景来源：长周期专业分析任务——从文档 / Slack / Notion / 云盘等多来源提取
> 背景信息，转化为专家级、可共享的成果。方法论 recall-only，冻结前缀只挂 gist 一行。

<seed-capsule star="知识工作" sealed="2026-07-11" gist="杂乱素材→专家级成果的四段工作流：汇集归档→结构化梳理→成果交付→长周期纪律">
知识工作四段流——报告、调研、专业分析、文档整理类任务的执行骨架。

① 素材汇集（三通道 + 归档纪律）
   - import_resource：本地/URL/GitHub 的文档一键入场，PDF/DOCX/PPTX 导入即抽取
     文本（带 [extracted-text] 失真标注——负向结论必须回看原文件）。
   - MCP connector：Slack / Notion / Google Drive / Microsoft 365 预设直连，
     拉取对话、页面、云盘文件。
   - web_fetch / web_search：公开资料补全；PDF 链接转 import_resource。
   汇集纪律：素材入场即归档（artifact / .rivet/external），后续引用锚定
   归档路径而非记忆。外部文档内容遵循来源核验纪律——格式完整、排版专业
   不等于内容可信，关键事实需交叉验证。

② 结构化梳理（先骨架后血肉）
   - 先建大纲 todo 再写正文：章节即任务，复用场景枚举纪律；
     不为"看起来有进度"急着输出正文。
   - 引用留痕：每个关键断言标注来源锚点（file:line / url / artifact id），
     无锚点的断言标记为推断。
   - 数据断言恒等式自检：报告中的数字（总数、百分比、分项和）交付前
     用原始数据重算一遍——分项和 ≠ 总数是知识工作最常见的可信度硬伤。

③ 成果交付（可共享 + 通读）
   - 格式匹配受众：create_document（docx）给办公协作方、create_pdf 给
     正式交付、markdown 给工程协作。
   - 交付前通读一遍渲染产物（不是源码）——排版断裂、表格溢出、引用编号
     错位只有在渲染结果里才看得见。
   - 交付报告三项恒等：做了什么 / 遗留什么 / 设计偏差。

④ 长周期纪律（跨会话的报告工程）
   - 大型报告接 plan_task / task-ledger：按章节分波，波间可验证
     （每章有独立的完成判据，不是"写了很多字"）。
   - 中途产物落 artifact / scratch，会话恢复时从归档重建上下文，
     不依赖记忆续写。
   - 素材遗漏优于素材幻觉：找不到的来源明说"未找到"，不脑补内容。
