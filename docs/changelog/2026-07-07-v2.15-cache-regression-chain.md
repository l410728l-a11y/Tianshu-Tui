# v2.15 缓存回归事故链 — 分支关联修复总览

日期：2026-07-06 ~ 07-07
分支：`feat/2.16-capability`
状态：全部已提交，回归测试钉住

## 事故起点

用户反馈 v2.15 会话成本显著高于 v2.13（同任务 3 元 vs 1.2 元）。Windows 会话 cache-log 显示 160 个请求里 158 个带 `frozenEvicted=true`，一条 20:39→21:23 的 `prefix_truncation` 链，最惨单轮 197K 输入只命中 8.7K（4.4%）、单次重建 18.9 万 token。本地 Mac 会话 `1924fe2b` 复现同样模式，确认非平台问题。

## 提交链（按因果序）

### 1. `d79d84f2` — 探针基建（先行铺设）

`feat(cache): prefix-divergence probe wiring`

engine 层 `prefixDiverged`（构建数组逐消息哈希）+ client 层 `wireDiverged`（最终上线字节哈希）双探针，写入 cache-log breadcrumb。本次事故链的两个 bug 都是靠这对探针定位的——engine 静默 + wire 报警 = 分歧在发送层变换；两层都报警 = engine 构建就变了。

### 2. `e0351f44` — 引入回归的提交（v2.15 发布内容）

`fix(cache): 前缀截断预防 — frozen 快照 commit 时机 + reasoning 字段归一化`

本意是修前缀截断，引入了 `frozenPendingMerged` 两段式 commit 机制，但边界 commit 的守卫依赖 `cachedFreshForUser`——该变量会被 `invalidateFreshCache()` 清空，而 `setIntentRetrievalRoute` 每条用户消息必触发 invalidate。invalidate 落在两轮之间 → pending 快照孤儿化，永远无法 commit → 历史消息每请求走 FATAL fallback、用当前 volatileBlock 重建 → 字节翻转 → `prefix_truncation`。

### 3. `85d9d9de` — 孤儿化根治

`fix(cache): frozen 快照孤儿化根治 — 边界 commit 改 pending 驱动清扫 + fallback 回存自愈`

- 边界 commit 改由 pending map 自身驱动（`buildOaiRequest` 前置清扫），去掉对可清空状态的依赖。
- fallback 重建结果回存 `frozenUserMerged` 自愈，杜绝每请求字节翻转。
- 曾试过 invalidate 时提前 commit，被测试否决（留陈旧中间版快照，破坏"历史槽位=最后上线字节"不变量）。
- 回归测试覆盖"invalidate 落在两轮之间"的生产序列。

详见 `2026-07-06-frozen-snapshot-orphan-fix.md`；约束条目已入 CLAUDE.md。

### 4. `3952cf12` — llm-speculation 缓存污染双修

`fix(api,agent): llm-speculation 缓存污染双修 — suffix 双写 copy-on-write + 侧路 usage 记账`

排查孤儿化修复后残留的两次 `wireDiverged idx 0`（system 消息"中途变字节"，本地会话 22:57）时发现的独立 bug 链，铁证是 prevCount/newCount 只差 1（正常主轮至少 +2，差 1 = spec 侧路请求混入探针基线）：

**Bug A：suffix 双写。** spec 侧路用 `[...request.messages, specUser]` 复用主请求消息对象引用；client `stream()` 里 system suffix 是原地 `content +=`。主轮已追加过一次，spec 重入再追加 → spec 请求 system = prompt+suffix+suffix → 整段 94K 前缀缓存 miss（~0.09 元/次）。修复：suffix 应用改 copy-on-write，任何 stream() 重入（spec 复用、FallbackStreamClient failover 重放）都不再污染调用方对象。

**Bug B：prefixProbe 泄漏。** `{...params.request}` 展开把主路径专属的 `prefixProbe: true` 带进 spec 请求 → 双 suffix 的 system 写入 wire 探针基线 → cache-log 出现幻影 `wireDiverged idx 0`。修复：spec 请求显式 `prefixProbe: undefined`。

**成本盲区收口（同提交）。** 全仓 6 个侧路 `stream()` 调用都用 `onStopReason: () => {}` 丢弃 usage，Bug A 的真金白银损失因此在日志完全隐形。新增：

- `createSidePathUsageRecorder`（loop-factory）：侧路 usage 入会话总量 + cache-log `event:'side_path'` 事件行。
- `SessionContext.addSidePathUsage`：只累计总量，不动占用估计锚点（`lastRealPromptTokens`/`tailEstimate`/校准比）——侧路测的是另一个消息数组，走 `addUsage` 会毒化压缩决策的占用感知。
- 接线 llm-speculation（kind=`llm-speculation`）与压缩总结两处（kind=`compact-summary`）。
- spec 引擎 fired/parseFailures/errors 计数落盘 meta `llmSpeculationEngine` 字段。
- 短 prompt 侧路（intent-router/goal-criteria/anti-anchoring/worker 修复）暂未接，sink 就位随时可加。

详见 `2026-07-06-llm-speculation-suffix-double-append.md`；约束条目已入 CLAUDE.md。

## 复盘要点

1. **守卫状态必须与生命周期匹配**：边界 commit 用了会被中途清空的变量做守卫，"同轮再 build 通过"掩盖了跨轮序列的失败。
2. **发送层变换禁止 mutation 调用方对象**：request 对象会被多个 stream() 重入，`content +=` 这类原地改写在重入时必然出错。
3. **展开语法复制请求时显式清除路径专属标志**：`prefixProbe` 泄漏毒化的是观测通道本身，产出误导性诊断。
4. **不记 usage 的 API 调用是成本盲区**：出问题只能靠账单倒查。任何新增侧路调用都应接 `recordSidePathUsage`。
5. **探针基建先行的价值**：`d79d84f2` 铺的双层探针让两个 bug 都在一天内从现象定位到根因。
