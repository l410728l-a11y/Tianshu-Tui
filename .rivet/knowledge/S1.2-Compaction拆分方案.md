# S1.2 拆分方案 — CompactionManager 独立类

> 接续 S1.1（纯函数已提取）。本文档给出 S1.2 的最终方案，供独立执行。
>
- **基础**：S1.2a 验证基线已提交（`074f0732d`），证明"独立类 + internal getter"可编译
- **关键约束**：JS `#private` 真私有，类外/declaration-merge/prototype 挂载都不可访问（TS18013 已验证）

---

## 一、为什么"零 getter"方案不可行

已验证排除的三条路：

| 方案 | 为什么不行 |
|------|-----------|
| 独立类 + 传 AgentSession 整体 | `#private` 类外不可见（TS18013） |
| declaration merging（`declare module` 加方法） | 实现不在原类体内，仍不可访问 `#private`（TS18013 已验证） |
| prototype 挂载（`Foo.prototype.added = ...`） | 同上，`#secret` 不可访问 |

**结论**：只要方法实现物理移出 AgentSession 类体，就必须通过某种公开接口访问私有状态。

---

## 二、最终方案：internal 聚合 getter（1 个 getter，非 80 个）

### 核心思路

不给每个私有成员加单独 getter（那会到 80 个），而是 **AgentSession 暴露 1 个 `internal` getter**，返回一个包含所有跨域访问器的对象。CompactionManager 写 `this.session.internal.advisorRuntime`。

```typescript
// agent-session.ts
/** S1.2: cross-class accessors for CompactionManager. Aggregated into one
 *  object so the internal surface is explicit and centrally managed. */
get internal(): AgentSessionInternal {
  return {
    advisorRuntime: this.#advisorRuntime,
    closeCodexProviderSessionsForHistoryRewrite: () => this.#closeCodexProviderSessionsForHistoryRewrite(),
    // ... 其余成员按需添加
  };
}
```

### 为什么这样比 80 个散落 getter 好

| 维度 | 80 个 internal_xxx | 1 个 internal 聚合 |
|------|-------------------|-------------------|
| AgentSession 公共面 | 膨胀 80 个成员 | 只增 1 个 getter |
| 集中管理 | 散落各处 | 一个对象，增删一目了然 |
| Manager 调用 | `this.session.internal_xxx` | `this.session.internal.xxx` |
| 后续收缩 | 删 80 处 | 删 1 个对象字段 |

## 五、实施步骤

### Step 1：定义 AgentSessionInternal 接口（compaction/internal-types.ts）

```typescript
/** 所有 CompactionManager 需要的跨域访问器。集中定义，便于审计和后续收缩。 */
export interface AgentSessionInternal {
  // 字段（readonly）
  readonly advisorRuntime: AdvisorRuntime | undefined;
  readonly agentKind: string;
  // ... 30 个字段
  // 方法
  readonly closeCodexProviderSessionsForHistoryRewrite: () => void;
  readonly emitSessionEvent: (event: AgentSessionEvent) => Promise<void>;
  // ... 50 个方法
}
```

### Step 2：AgentSession 加 internal getter

```typescript
get internal(): AgentSessionInternal {
  return this.#internalCache ??= {
    advisorRuntime: this.#advisorRuntime,
    // ...
  };
}
#internalCache?: AgentSessionInternal;  // 缓存避免每调用重建
```

注意：内部方法用箭头函数闭包绑定 this，缓存后调用安全。

### Step 3：CompactionManager 搬入 A 类方法（共 20 个）

从 agent-session.ts 物理剪切 20 个方法到 compaction-manager.ts。每个方法内：
- `this.#xxx` → `this.session.internal.xxx`（跨域）
- `this.#aMethod()` → `this.aMethod()`（A 类互调，搬到 Manager 后是普通方法）
- `this.publicXxx` → `this.session.publicXxx`（公共成员）
- 纯函数（S1.1）→ 直接调 `projectSnapcompactContextTokens(...)` 等

### Step 4：AgentSession 保留委托存根

```typescript
// 原 dropImages 改为委托
async dropImages(): Promise<{ removed: number }> {
  return this.#compactionManager.dropImages();
}
```

5 个公共 API 方法（compact/handoff/shake/dropImages/runIdleCompaction）必须保留委托存根（公共签名不变）。

## 七、风险与缓解

| 风险 | 缓解 |
|------|------|
| internal 对象缓存后，字段值过期（如 #promptGeneration 变了） | **字段类必须实时读，不能缓存**。internal getter 每次重建对象（无 #internalCache），或字段用 getter 不缓存值 |
| 方法闭包绑定 this 错误 | 全部用箭头函数 `(...) => this.#xxx(...)` |
| 12 个 A 类互调搬到 Manager 后改名冲突 | A 类方法去掉 `#` 前缀作为 Manager 普通方法，不会冲突 |
| 搬运遗漏导致 AgentSession 残留死代码 | S1.4 删除时 grep 确认无调用 |

### 关于字段过期（最重要）

`#promptGeneration`、`#retryAttempt`、`#autoCompactionAbortController` 这类**频繁变化的状态字段**，如果 internal 对象缓存了快照，会读到旧值。两种解法：

**解法 A（推荐）**：internal getter 不缓存，每次返回新对象（字段读取时 `this.#xxx` 是实时值）
```typescript
get internal(): AgentSessionInternal {
  return {
    get promptGeneration() { return thisRef.#promptGeneration; },  // getter 才实时
    // ...
  };
}
```
但这要存 `thisRef`，略繁琐。

**解法 B**：字段全部用 getter，方法用闭包
```typescript
get internal(): AgentSessionInternal {
  const self = this;
  return {
    get advisorRuntime() { return self.#advisorRuntime; },
    get promptGeneration() { return self.#promptGeneration; },
    closeCodex: () => self.#closeCodexProviderSessionsForHistoryRewrite(),
    // ...
  };
}
```
每次访问 `internal.promptGeneration` 实时读 `self.#promptGeneration`。每次调 `internal.xxx()` 都重建对象——但对象很轻（几十个引用），成本可忽略。

**用解法 B**。绝不缓存 internal 对象（`#internalCache` 会冻结字段快照）。
