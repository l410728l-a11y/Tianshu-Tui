# 诊断策略切换框架

> 2026-06-19 从 GLM flushToolCalls 排查中蒸馏。来源文档：`debug-glm-flush-tool-args-loss.md`。
> 与 `debug-t7-collapse-cache-cliff.md`（数据诊断铁律）互补——那篇讲数据怎么读，这篇讲什么时候换策略。

## 核心命题

静态推理有边界。当两个已验证事实互相矛盾时，继续读代码的边际收益趋零。此时正确的做法不是「再读一个文件看看」，而是写最小复现测试直接驱动疑似函数——3 行探针比 6 个文件的推理链更有决定性。

## 五条策略纪律

### 1. 悖论是策略切换信号，不是深度推理邀请

当 A 和 B 都被独立验证为真但逻辑上互斥（如「落盘的 arguments 完整可 parse」vs「工具收到空 input」），说明推理链断了——有一条中间环节在你视线之外。继续沿着眼睛能看到的路径推理不会找到它。

**触发动作：** 停止读文件，写复现测试驱动疑似函数。

### 2. 诊断循环里测试在「改」之前

开发循环：读 → 改 → diff → test（测试验证修复）

诊断循环：读现象 → 读疑似代码 → 写复现测试 → RED 锁定根因 → 改 → GREEN 验证修复

关键差异：诊断循环里复现测试是**定位工具**，不是验证工具。不要等改完才写测试——那时你已经没有物理证据确认根因了。

### 3. "exposed for testing" 是最高优先级入口

当源码标注一个函数「exposed for testing」时，作者已经为你留好了单元级复现接口。排查时优先用它构造最小用例，而不是绕过它追整条数据流。

案例：`openai-client.ts` 的 `processDelta` 标注了 "exposed for testing"，外部会话 3 分钟锁定根因的关键一步就是直接喂 5 个 chunk 序列用例驱动它。

### 4. 静默 catch { return default } 是诊断黑洞

一行 `catch { input = {} }` 同时制造了 bug 和掩盖了 bug。排查方法论：

- 追数据流时优先 grep 所有静默 catch 分支：`catch\s*\{[^}]*\}`
- 任何 catch 分支如果只是静默替换默认值而不打日志，视为高风险
- 优先怀疑静默 catch 是数据消失点

### 5. 安全机制在 bug 链路上容易被误读

doom loop 检测器准确拦截了循环——它没问题。真正的问题是上游 `flushToolCalls` 制造了无法自行恢复的失败循环。排查时先沿链路往上走：**谁制造了触发条件？** 不要盯着拦截器问「你为什么拦我」。

## 诊断决策树

```
遇到问题
│
├─ 有悖论？（两个已验证事实互相矛盾）
│  └─ 是 → 停止读文件，写最小复现测试驱动疑似函数
│
├─ 无悖论 → 读相关代码
│  │
│  ├─ 疑似路径上有 "exposed for testing" 的函数？
│  │  └─ 是 → 优先用它构造复现
│  │
│  ├─ 同一函数追了 3+ 文件仍未定位？
│  │  └─ 是 → 强制切换复现测试模式（硬性触发）
│  │
│  └─ 有静默 catch { return default }？
│     └─ 是 → 先怀疑它，打日志验证
│
├─ 看到安全机制（doom loop / 降级 / 拦截）？
│  └─ 先追上游：谁制造了触发条件？
│
└─ 定位到根因 → 修 → 复现测试 RED→GREEN → 回归测试
```

## 验证实例

GLM flushToolCalls 排查中两条路径的对照：

| 行为 | 天枢（6 文件卡死） | 外部会话（3 分钟） |
|------|-------------------|-------------------|
| 读 JSONL 发现悖论 | ✓ | ✓ |
| 读代码排除路径 | 4 条（6 文件） | 4 条 |
| 注意到 "exposed for testing" | ✗ | ✓ |
| 写复现测试驱动 | ✗ | ✓（5 个 chunk 序列） |
| 锁定根因 | ✗ | ✓ |

差距不在能力，在行为约束：天枢当时的 evidence-scope 规则把「查证」等同于「读文件」，手段列表里没有「写测试做实验」。该规则已在此次排查后修复（`cd7030d2`），加入了诊断策略切换条款。
