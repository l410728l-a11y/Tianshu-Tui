# 计划—执行方法论：蒸馏自 sandbox-path-grants 实战

> 蒸馏日期：2026-06-14
> 来源：对 `sandbox-path-grants_f68448d2.plan.md` → 九文件落地 的完整审查
> 用途：后续复杂 spec 计划与执行可复用此方法论

---

## 核心发现

该计划—执行过程展现了**100% 完成率**的关键不在于执行者特别勤奋，而在于**计划本身的结构消除了执行阶段的歧义**。计划写完时，所有设计决策已经闭环——执行只是翻译。

> **功劳归属**：sandbox-path-grants 的 100% 完成率，是计划质量的结果，不是执行者个人能力的结果。计划把双门同步、威胁模型、反证测试全部前置消化——执行阶段几乎没有自由度可以做错。这意味着 100% 是可复制的：它绑定在方法上，不绑定在执行者身上。感谢 Cursor Opus 4.8 作为执行当事人确认了这一判断。

## 五阶段推理流水线

```
问题建模 → 边界标定 → 最小侵入点选择 → 双门对齐 → 测试反证收敛
```

每阶段的产出物和反模式：

| 阶段 | 产出物 | 跳过此阶段的后果 |
|------|--------|-----------------|
| 问题建模 | 系统诊断（N 个独立门共享同一缺口），引用真实文件路径和函数名 | 方案只修了一个门，另一个门仍然硬拒绝 |
| 边界标定 | "不改什么"的显式列表 | 执行中方案膨胀，改到计划外文件 |
| 注入点选择 | 每个改动点的"当前行为→改后行为→为什么安全" | 执行者自行选择注入位置，可能与设计者意图偏离 |
| 双门对齐 | 数据流图 + 语义约束（write 蕴含 read） | 两个门行为不一致，出现可通过文件工具但 bash 被阻的裂缝 |
| 测试反证 | 偷懒实现会红的测试断言 | 测试全部 GREEN 但实现有隐蔽缺口 |

## 六条可复用原则

### 原则 A：双门模型

> 当一个系统有两个独立执行路径共享同一 enforcement 目标时，方案必须让两个门同步感知同一授权状态。单门方案不是简化——是不完整。

**适用场景**：任何涉及多个独立 enforcement point 的变更（安全、权限、沙箱、验证）。

**检查方法**：grep 共享 enforcement 状态的导入方（不是 guard 函数的调用者——两个门可能不共享函数，只共享状态模块），或枚举"这个操作的所有拒绝点"（沿着操作往下数门，而非沿着函数往上数调用者）。

**实证**：path-grants 场景中，`path-validate.ts` 和 `sandbox-profile.ts` 不共享 guard 函数（sandbox 不调用 validatePathSafe），但共享同一个状态模块 `path-grants.ts`。grep `validatePathSafe` 调用者返回一批文件工具与管线，但**不含 sandbox-profile.ts**——沙箱门不调用该 guard，所以 grep 调用者必然漏掉它。grep `path-grants` 导入方则两个门都出现。教训：**枚举执行点，不枚举调用者**。

### 原则 B：存储模式跟随既有先例

> 新模块的存储模式刻意镜像代码库中已有的模式。在同一代码库中，一致性高于"最佳实践"。

**实例**：`path-grants.ts` 的模块级 `_grants` 数组 + `_resetForTest()` 镜像 `sandbox-profile.ts` 的 `_cachedActiveBackend` 模式。

**反模式**：引入 DI 容器、事件总线、或 class 封装——当代码库中已有模块级可变状态的先例时，引入新模式增加认知负荷。

### 原则 C：每调用重建 > 事件总线

> 当消费者每次调用时重建状态比维护精确的失效传播更可靠时，选择重建。

**实例**：`defaultWritableRoots()` 每 bash 命令调用时重建，所以新授权不需要 invalidate 信号即可生效。

**判断标准**：重建成本低于失效传播的复杂度时，选重建。`defaultWritableRoots` 每次调用只是遍历一个数组，成本极低。

### 原则 D：错误信息包含修复路径

> 错误信息不描述问题，描述**下一步行动**。

**实例**：从 `"Path outside project directory: <p>"` 升级为 `"Path outside project directory: <p>. If the user authorized this, call request_path_access (or approve the prompt) to grant <mode> access to this path."`

**模板**：`"<what's wrong>. <what to do about it>."`

### 原则 E：安全注释即文档

> 代码中的安全注释精确描述威胁模型，不是装饰。

**必须包含的信息**：
- 具体威胁（macOS `/var→/private/var`、符号链接逃逸、前缀泄漏）
- 为什么这个检查是必要的（不做的后果）
- 什么场景下这个检查会触发

**反模式**：`// security check` — 没告诉维护者为什么不能删。

### 原则 G：方法论文档也是代码——可执行指令必须经实证检验

> 方法论文档中的 grep 命令、正则表达式、心智操作步骤，与代码中的函数实现具有相同的性质——它们都是可执行的、可验证的、可能出错的指令。**写完一条 grep 示例而不跑它，等于写完一个函数而不跑它的测试。**

**实证**：轻量模板中写了一条 `grep "validatePathSafe"` 来找"第二扇门"，但它漏掉了 sandbox-profile.ts——因为两个门的关系不是 caller/callee，而是并行执行点共享状态模块。跑一遍真实的 grep 就能当场发现这个错误，但我们在"写作模式"里评估它"看起来对"。

**检查方法**：方法论文档中每出现一条可执行指令（grep、regex、shell 命令），必须在真实代码库中跑一遍并确认输出覆盖预期目标。不跑不提交。

## 执行模式特征

### 1. 全量覆盖，不挑拣
计划列了 8 个 todo，实现覆盖了全部 8 个。没有 P0/P1 优先级降级——因为"双门必须同时改"这个约束决定了降级任何一个等于方案不成立。

**推论**：如果方案有硬依赖（"A 没做则 B 也没意义"），不要在计划阶段拆 P0/P1——要么全做，要么重设计方案让各部分独立。

### 2. 测试与源码同时交付
5 个测试文件对应 5 个实现改动点，同时出现在 untracked/modified 列表中。不是"实现完再补测试"。

**推论**：测试是设计的一部分，不是事后验证。写测试的过程会发现设计漏洞。

### 3. 提示词同步更新
`prompt/static.ts` 加了一行指引，让模型知道"工作区外路径怎么处理"。不是事后想起的。

**推论**：如果改动引入了新的工具或新的行为模式，提示词必须在同一批次更新。否则模型不知道新能力存在。

### 4. 先例引用降低认知负荷
计划中显式引用了 `learnBashPrefix` 和 `_cachedActiveBackend` 作为先例。

**推论**：审查者（和人）理解新代码的成本取决于它与已有代码的相似度。引用先例是最低成本的文档。

## 计划自检清单（交付天梁域前）

- [ ] 所有文件路径引用真实存在（不是"某个文件"）
- [ ] 数据流图覆盖所有触发路径
- [ ] 显式列出了"不改什么"
- [ ] 每个改动写清楚了"为什么安全"
- [ ] 安全不变量逐条有测试对应
- [ ] 测试表包含反证断言（偷懒实现会红）
- [ ] 引用了代码库中已有的类似模式作为先例
- [ ] 执行次序按依赖排序，每个 wave 独立可提交
- [ ] 提示词变更与代码变更在同一批次

## 已知局限

1. 此方法论适用于**多门协同变更**（安全/权限/沙箱类）。对于单模块内聚重构（如 tool-group 生产化），使用[轻量版模板](docs/superpowers/plans/2026-06-14-plan-methodology-lightweight.md)——缩减为五阶段，去掉触发路径清单和安全不变量（降为可选附录），新增消费者影响清单作为阶段二的核心。

2. 五阶段是**顺序依赖**的——跳过问题建模直接选注入点，等于在不知道全部约束的情况下做局部决策。但实践中允许迭代：如果注入点选择阶段发现新约束，回退更新问题建模。

3. 此方法论假设**设计者和执行者是不同角色**。如果同一人设计并执行（如 sandbox-path-grants 原案例），部分结构（如"不改什么"的显式声明）可能被压缩为设计者的内在判断。但拆分角色时必须显式化。
