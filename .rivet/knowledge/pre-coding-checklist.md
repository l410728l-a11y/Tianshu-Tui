# 编码前四道扫描

> 来源：f13b0b82 → e5bdd033 的缺陷归族分析。f13b0b82 引入 contextCalibrationRatio 校准机制，有三个缺陷在生产中暴露（首轮 ratio 毒化、无 clamp、α=1 首次特例不可恢复）。e5bdd033 用三道防御修复。本文件提取"开发前"能拦截这类缺陷的通用检查，补充天权规划域（现有覆盖空间结构：文件依赖/调用方，缺时间结构和参数验证）。

## 缺陷族：把运行时代码当纯函数推理

f13b0b82 的实现者把 `ratio = apiTokens / localEstimate` 当数学公式写，忽略了输入字段有自己的就绪时间线。五个表面 bug 的底层是同一个盲区，展开为四层认知缺失：时序推理缺失（首轮 prefixOverhead 未就绪）、哨兵值复用（ratio=1 双重语义）、反演验证缺失（EMA α=0.7 对 10000x 毒化无效）、测试 fixture 遮蔽（setUp 跳过未初始化窗口）。

## 四道扫描

### 1. 字段就绪时序扫描（Temporal Readiness Scan）

**问题**：函数消费多个字段的组合，但这些字段的初始化不同步。函数在"部分字段已就绪、部分尚未"的窗口被调用时，输出未定义。

**f13b0b82 案例**：`localEstimate = estimatedTokens + prefixOverhead`。estimatedTokens 从构造函数开始累加，prefixOverhead 要等 `ensurePrefixOverhead`（在 `maybeCompact` 内）跑完才赋值。addUsage 在首轮（ensurePrefixOverhead 之前）被调用时，localEstimate 缺了 30K+ tokens 的 system prompt + tool schemas，ratio 被毒化到 10000x。

**检查方法**：列出函数每个输入字段的赋值时间线，标出函数被调用的时刻，检查是否存在交叉但不重叠的窗口。30 秒能做完。

```
字段              | 构造时 | addUserMessage | ensurePrefixOverhead | addUsage(API返回)
estimatedTokens   |  =0    | +=msg tokens   | —                    | —
prefixOverhead    |  =0    | —              | =N                   | —
                  |        |                |                      |
addUsage 调用时刻 → T1(首轮): overhead=0 ✗    T2(后续): overhead=N ✓
```

**规则**：如果函数 F 在字段 A 已就绪但 B 尚未的窗口中被调用，F 必须有 defer 守卫（检测 B 未就绪并跳过/降级），或保证 F 不在该窗口被调用。

### 2. 哨兵值审毒（Sentinel Poisoning Audit）

**问题**：用业务域的合法值标记未初始化状态。业务值会被自然产生，与状态标记冲突。

**f13b0b82 案例**：`alpha = contextCalibrationRatio === 1 ? 1 : 0.7`。ratio=1 的物理含义是"本地估算完美等于真实值"——校准的最佳结果。用它判断"是否首次校准"，等于说"完美的校准等于没校准"。如果某轮 API 返回恰好等于 localEstimate（ratio 真的=1），后续所有校准都错误走 α=1 全量覆盖。

**检查方法**：grep `=== <常量>` 或 `!== <常量>` 形式的条件分支，问"这个常量是否同时也是业务域的合法输出？"如果是，分支有被业务值误触发的风险。

**规则**：区分"已初始化"和"未初始化"不能用业务域合法值。用 boolean（如 `hasCalibrated`）或 nullable。值的语义角色要唯一——一个数字不能同时是"状态标记"和"业务结果"。

### 3. 参数化算法的极端值手推（Worst-Case Walk-Through）

**问题**：引入有状态参数（α、阈值、窗口大小）的算法，注释声称"平滑/鲁棒"，但声称未经数值实验验证。参数在正常输入下表现良好，在极端输入下完全失效。

**f13b0b82 案例**：EMA α=0.7 的注释说"not jittered by one outlier"。但 α=1 首次特例 + ratio=10000 的组合下，首次校准全量写入 10000x，后续 α=0.7 需 19 轮才能拉回 1x（0.3^19 ≈ 1/10000）。EMA 对 2x 波动是平滑的，对 10000x 毒化完全无效。

**检查方法**：写代码前用该算法能产生的最极端输入手推一轮——不是典型输入，是最坏情况输入。问"如果参数取极端值，算法行为是什么？"如果答案和注释声称矛盾，要么改参数，要么加守卫。

**规则**：任何算法的参数选择，必须有极端值手推作为支撑。参数选择的理由是物理事实（手推数值），不是脑补的模型（"应该是平滑的"）。

### 4. 测试 fixture 与生产初始化序列对齐（Test Fixture Alignment）

**问题**：测试 setUp 用特定顺序预设了所有依赖项，恰好跳过了生产环境中最先触发的路径——依赖项尚未就绪的窗口。测试绿灯只能证明"在预设条件下正确"，不能证明"在生产初始化序列中安全"。

**f13b0b82 案例**：所有校准测试以 `ctx.setPrefixOverhead(10_000)` 开头再调 `addUsage`。但生产中 `addUsage`（API 返回 usage）在首轮先于 `setPrefixOverhead` 执行。测试覆盖了"prefixOverhead 已设置"的世界，没覆盖"prefixOverhead=0"的首轮路径。这个首轮路径恰好是 ratio 被毒化到 10000x 的触发点。

**检查方法**：列出函数在生产中的实际调用顺序（不假设所有 setUp 已完成），与测试的 setUp 顺序对比。问：生产中有没有测试没覆盖的"字段未就绪"调用窗口？

```
生产时序（首轮）:
  construct → addUserMessage → [addUsage 在此被调] → ensurePrefixOverhead
  ↑ 此时 prefixOverhead=0，localEstimate 缺了 30K+ tokens

测试时序（所有用例）:
  construct → setPrefixOverhead(10_000) → [addUsage 在此被调]
  ↑ prefixOverhead 已就绪，从不触发未初始化路径
```

**规则**：测试 setUp 必须镜像生产的最早调用窗口，而不是"所有依赖项就绪后"的理想世界。如果生产中存在函数在字段 X 未就绪时被调用的路径，测试必须至少有一个用例复现该路径。

## 应用顺序

这四道扫描在天权规划域"先读完再规划"之后、动手写代码之前执行：

1. 先读完代码空间结构（现有：天权 scope check）
2. **时序扫描**：函数输入字段在调用时刻都就绪了吗？
3. **哨兵审毒**：用来判断状态的值会被业务值冒充吗？
4. **极端值手推**：参数在最坏输入下还满足注释声称吗？
5. **fixture 对齐**：测试 setUp 覆盖了生产的最早调用窗口吗？
6. 动手写代码

## 适用条件：不是所有代码都需要四道扫描

这四道扫描的共同根因是「把运行时代码当纯函数推理」——忽略了状态机的时序和初始化语义。如果你的代码不涉及这个盲区，扫描可以跳过或快速通过。

### 触发决策图

```
你写的新代码是否消费了多个字段的组合？
  ├── 否 → 四道扫描全部跳过（纯函数、简单 getter、常量定义等无关）
  └── 是 → 这些字段的初始化是否同步完成？
        ├── 是（同构造函数、同一次赋值）→ 扫描 1（时序）快速通过，继续
        └── 否（不同生命周期阶段赋值）→ 扫描 1 必须执行

你写的新代码是否用某个常量值判断"是否首次/是否初始化"？
  ├── 是 → 扫描 2（哨兵审毒）必须执行
  └── 否（用了独立 boolean / nullable）→ 扫描 2 通过

你写的新代码是否引入了有状态参数（α、阈值、窗口大小、衰减系数）？
  ├── 是 → 扫描 3（极端值手推）必须执行
  └── 否 → 扫描 3 跳过

你是否在写测试？
  ├── 是 → 扫描 4（fixture 对齐）必须执行
  └── 否 → 扫描 4 跳过（实现阶段只跑前三道）
```

### 每道扫描的适用边界

| 扫描 | 触发条件 | 不适用的情况（可跳过） |
|------|---------|---------------------|
| 1. 时序扫描 | 函数消费多个字段，字段在不同生命周期阶段赋值 | 纯函数、单字段输入、所有字段同步初始化、无状态工具函数 |
| 2. 哨兵审毒 | 代码中存在 `=== 常量` 或 `!== 常量` 形式的条件分支，用于判断状态 | 用独立 boolean/nullable 表示状态、不涉及初始化判断 |
| 3. 极端值手推 | 引入有参数（α/阈值/窗口）的算法，且注释声称"平滑/鲁棒/抗噪" | 无参数算法、参数有 proven bound（如来自数学推导而非经验选择） |
| 4. fixture 对齐 | 写测试，且测试 setUp 预设了依赖项的特定顺序 | 纯函数测试（无状态 setUp）、测试 setUp 完全镜像生产调用链 |

### 本缺陷族之外：这四道扫描拦不住什么

明确标出边界——以下常见缺陷不在四道扫描的覆盖范围内：

- **纯函数逻辑错误**（如 `a / b` 写成 `b / a`）：时序扫描不会触发，因为没有多字段初始化问题
- **并发竞态**（两个异步操作的结果到达顺序不确定）：时序扫描只覆盖初始化顺序，不覆盖运行时交错
- **资源泄漏**（文件句柄、定时器未清理）：不涉及值语义或参数选择
- **性能退化**（O(n²) 意外产生）：极端值手推覆盖参数边界，不覆盖输入规模边界
- **安全漏洞**（注入、越权）：需要独立的安全审查，不在本扫描范围内

这些缺陷族需要各自的检查方法。四道扫描专攻一类问题——状态机时序和值语义——不应膨胀为通用质量门。

## 与现有体系的关系

| 现有理论 | 覆盖的阶段 | 本文件补充的阶段 |
|---------|-----------|----------------|
| 天权：先读完再规划 | 规划（空间结构） | 规划（时间结构 + 参数验证） |
| 瑶光：复现即证 | 验证（bugfix RED→GREEN） | 开发前（极端值手推） |
| 天府：结构是承诺 | 设计（数据结构 fail-closed） | 设计（值语义角色唯一） |
| TDD 可行性探针 | 测试（机制可行性） | 测试（fixture 与生产初始化序列对齐 — 第四道扫描） |

共同特征：这四道扫描都是"开发前"检查，不是"开发后"验证。现有胶囊体系在验证阶段很强（瑶光的复现纪律），但在规划阶段缺了这四道。第四道扫描（fixture 对齐）桥接了"开发前检查"和"测试编写"——在写测试 setUp 之前就检查生产调用序列，不让 fixture 预设顺序遮蔽未初始化路径。
