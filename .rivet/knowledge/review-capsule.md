# 审查胶囊 — 自审方法论

> 从 p1-trio-review、toolcard-budget-review、budget-unify-review 三轮审查中蒸馏。
> 每条规则都有对应的真实 bug 作为证据（不是假设）。
> 用于：实现完成后、deliver_task 之前，按此清单自审。

---

## 🔍 六大审查模式

### P1: 双源预算脱节

**现象**：两个独立模块各自计算同一类资源（行数、额度、配额），但互不知道对方的实际消费值。预算方按值 A 预留，消费方用值 B 渲染，A ≠ B → 溢出。

**检测方法**：
1. 找到系统中所有 `limit / budget / max / cap` 计算
2. 追踪每个计算值被谁消费
3. 确认：预算计算用的值 = 渲染消费用的值（精确匹配，不是"差不多"）

**实证**：stream.tsx 用 perToolBudget=8 预算，tool-card.tsx 用 cappedCollapsedLines=15 渲染，差额 7 行/卡，3 卡溢出 21 行。

**修复模式**：提取 shared budget module，单一数据源，双方都从同一函数获取值。

---

### P2: 参数化测试盲区

**现象**：测试矩阵覆盖面看似全面（N×M 个用例），但某个关键维度被硬编码为常量。测试全绿但覆盖范围外有 bug。

**检测方法**：
1. 检查每个 `for` 循环和参数化测试的边界
2. 问：哪个维度被写死了？为什么？
3. 特别关注：条件组合中的"默认值"（如 `computeBudget(rows, 1)` 中的 `1`）

**实证**：48 个测试覆盖 rows×nCards 的折叠态不变式，但展开态测试写死 `n=1`，导致展开+多卡组合从未被测试。

**修复模式**：每个维度都参数化。如果组合爆炸，用分层——先测基础不变式（每维度独立），再测交叉不变式。

---

### P3: 三态声明两态实现

**现象**：类型/文档声明了 N 个状态，但只有 M 个（M < N）被实际使用或可达。多余的状态是死代码，误导读者。

**检测方法**：
1. 对每个枚举值/联合类型成员：grep 赋值点（set/构造），grep 使用点（switch/if）
2. 如果某个值只有类型声明但没有赋值点 → 死状态
3. 如果某个值有赋值但没有效果 → 死逻辑

**实证**：`PlanModeState = 'off' | 'planning' | 'approved'`，但 `approved` 从未被 `setPlanModeState` 设置——`/plan-approve` 实际调用 `exitPlanMode()` 设为 `off`。

**修复模式**：删除死状态。类型定义 = 运行时可达状态。如果将来需要，YAGNI 以后再加。

---

### P4: 幽灵参数

**现象**：函数签名接收了参数，但函数体完全不引用它。调用方以为参数在生效，实际被忽略。

**检测方法**：
1. 对每个函数参数：在函数体内搜索其标识符
2. 如果只在签名中出现 → 幽灵参数
3. 特别警惕"收了 count/n 但公式里只有硬编码常数"

**实证**：`computeBudget(termRows, liveToolCount)` 中 `expandedLines = min(15, max(cardLines, available - 1))`——`liveToolCount` 完全没出现在公式中。

**修复模式**：每个参数必须影响至少一个输出值。如果不需要，删参数。

---

### P5: 未接线的加载器

**现象**：实现了 load/parse/register 函数，但启动流程从不调用它。功能声明为"支持"但实际不工作。

**检测方法**：
1. 对每个 `load*` / `register*` / `init*` 函数：grep 调用点
2. 排除定义处和测试处，看生产代码中是否有调用
3. 如果为 0 → 未接线

**实证**：`loadFromDirectory` 在 profile-registry.ts 中完整实现，有测试，但 main.tsx 启动时从未调用——用户自定义 profile 永远不加载。

**修复模式**：接线到启动流程，或如果当前不需要就删除实现（YAGNI）。

---

### P6: 门禁白名单过宽

**现象**：安全/权限白名单包含了本应被禁止的操作。白名单看似合理（工具名不含"write"），但间接能力可绕过限制。

**检测方法**：
1. 对白名单中每个条目：列出其可达操作（不是表面功能，而是能触发的副作用）
2. 特别警惕 `delegate_*` / `deliver_*` / `eval` 类——它们能间接执行写操作
3. 问：如果这个工具是唯一的攻击面，能造成什么损害？

**实证**：`PLAN_MODE_ALLOWED_TOOLS` 包含 `delegate_task`——虽然名字不含"write"，但 delegate 派出的 worker 有完整写权限，完全绕过 plan-mode 只读限制。

**修复模式**：最小权限原则——白名单只含纯读、无派生副作用的工具。有间接执行能力的工具一律排除。

## 📋 自审清单（deliver_task 前）

实现完成后，逐项检查：

- [ ] **P1 预算一致性**：如果涉及资源分配，预算计算方和消费方用的是同一个函数/值吗？
- [ ] **P2 测试维度**：参数化测试的每个维度都覆盖了吗？有没有硬编码的常量？
- [ ] **P3 状态完整性**：每个声明状态都有赋值路径吗？每个赋值路径都有对应的使用者吗？
- [ ] **P4 参数活性**：每个函数参数都在函数体内被引用了吗？
- [ ] **P5 接线验证**：每个 load/register/init 函数都被启动流程调用了吗？
- [ ] **P6 最小权限**：白名单中的每个工具，它的间接可达操作是什么？有绕过风险吗？

## ⚠️ 元教训

1. **"verified" 不等于 "correct"**：天枢声称"48 测试零溢出"是真的——在它测的范围内。范围外就漏了。
2. **手算不可替代**：审查者用 node 精确计算各终端尺寸，发现公式输出与声明不符。自动化测试只验证你想到的组合。
3. **参数化公式要手动验算**：写了公式后，用 node -e 算一遍边界值，不要只跑测试。
4. **审查者的修复也可能是错的**：审查者给的 expandedLines 公式也不对（n=1 时跟原 bug 一样）。修复必须独立验算。
