# 计划语义自检门 (Plan Semantic Gate)

> **产生背景**: 2026-06-25 编辑/压缩工具优化计划中,三个实现缺陷(P1a 归一化悖论、P1b 正则失配、P2 索引外推)的根因是同一模式——计划写完后没有对每条改动的伪代码做物理推演。格式完整(行号精确、9阶段齐全)但语义理解停留在符号层面。

此约束嵌入计划生成流程,要求计划在交付前通过三条物理验证——不写代码,只留推演痕迹。

---

## 规则一: 每条算法改动用真实数据手推一轮

**触发条件**: 计划中包含任何新函数 / 修改现有函数逻辑 / 数据转换。

**要求**: 在改动点明细表的每条改动后,追加一行"自检输入→期望输出"。输入必须是具体字节(不是类型签名),输出必须是具体字符串/结构。

**格式示例**:
```
| 1 | edit.ts | fuzzy 命中后调用 diffBlock | 自检: old="\tindented line" vs act="    indented line" → 期望 diff 包含 "\t" (tab) 和 "    " (spaces) 的可见差异 |
```

**反模式**: `自检: 输入任意两个字符串,输出 diff 格式` ← 这是类型签名,不是自检。必须给具体内容。

**判断标准**: 执行者不读原代码,只看自检输入,就能手写出期望输出并验证它和伪代码一致。

---

## 规则二: 正则表达式必须配来源验证

**触发条件**: 计划中任何正则表达式(包括 `new RegExp`、字符串 `match`、`split` 等模式匹配)。

**要求**: 正则旁边用注释标注:
1. 它匹配的真实文本片段(从目标文件/输出中逐字复制)
2. 该文本的来源(`src/foo/bar.ts:L330` 或 `node --test 输出` 等)

**格式示例**:
```
/exit code[:\s]+(\d+)/i  // 来源: src/tools/run-tests.ts:L330 formatOutput → "Exit code: 1"
```

**反模式**: 正则后没有注释,或注释写 `// matches exit code`(没有真实输入验证)。

---

## 规则三: 合并/去重提案必须列出差异矩阵

**触发条件**: 计划中提到"A 和 B 功能重复,合并为 C" 或 "提取共享函数 X"。

**要求**: 在方案描述后,用 4 列表格对比被合并模块的差异:

| 维度 | 模块 A | 模块 B |
|------|--------|--------|
| 遍历方向 | (正向/反向/具体描述) | |
| 索引语义 | (存的是什么,含义) | |
| 工具集 | (处理哪些工具名) | |
| 边界标记 | (wasTruncated / LAG / 其他) | |

**判断**: 任一维度不同 → 禁止在该维度上合并。只能提取逐字符相同的纯函数。

**格式示例**(来自 staleness-detect vs agent-diet):
```
| 维度 | staleness-detect | agent-diet |
|------|-----------------|------------|
| 遍历方向 | 反向 i+1 索引 | 正向 tool_call_id 反查 |
| 索引语义 | assistant.tool_calls 存 i+1(tool result) | tool msg 通过 extractPath 存 i(自身) |
| 工具集 | read_file, grep | read_file, grep, glob |
| 边界标记 | 无 | wasTruncated, supplementary |
→ 结论: 禁止合并 fileReads 构建。只提取 rangeContains + parseOptionalInt 两个纯函数。
```

---

## 嵌入点

此三条规则在计划生成流程中嵌入位置:

```
阶段三(注入点选择) 完成后 → 对每条算法改动执行 规则一
阶段三结束后 → 对全文中所有正则执行 规则二
阶段三"为什么安全"段 → 若有"合并 A 和 B"提议,执行 规则三
```

模板自检清单追加一项:
- [ ] **语义自检**: 算法改动有自检输入/输出,正则有来源注释,合并提案有差异矩阵

---

## 为什么要"不写代码"

这三点只需要 grep + read_file + 手推——不需要启动测试框架。它们验证的不是代码的正确性,而是**计划中伪代码与物理世界(文件内容、输出格式、模块数据流)的一致性**。

代码的正确性由执行者的 TDD 验证。计划的自检验证的是"这个伪代码描述的功能,在真实输入上跑会产出什么"——这是测试写不出来的前置检查。
